# SQL проекты

## Проекты в Greenplum:

### 1. `GP_udf_delta_upsert` — создание UDF для загрузки данных по ключу, реализация метода delta upsert.

**Проблема**: При ежедневной загрузке данных в витрину возникали дубликаты записей по ключу, а полная перезагрузка таблицы занимала >15 минут.

**Решение**: Реализовала функцию `f_load_du()` на PL/pgSQL, которая:
- Группирует данные из staging-таблицы по месяцу (ключ партиционирования)
- Удаляет старые записи за целевой месяц
- Вставляет актуальные данные только за этот период
- Автоматически собирает статистику (`ANALYZE`) после загрузки

**Стек**: Greenplum, PL/pgSQL, Dynamic SQL, Partitioning

**Результат**:
- Время загрузки сократилось с 15+ минут до ~3 минут за счёт инкрементального обновления
- Исключены дубликаты по бизнес-ключу
- Функция переиспользуется в 3+ DAG'ах Airflow

### 2. `GP_udf_delta_partition` — создание UDF для загрузки данных с помощью подмены партиций, реализация метода delta partition.

**Проблема**: Большие факт-таблицы (миллионы строк) требовали быстрой инкрементальной загрузки без блокировки всей таблицы.

**Решение**: Создала функцию `f_load_delta_partition()`, которая:
- Принимает дату загрузки (`{{ ds }}`) и ключ партиционирования
- Загружает данные только за целевую дату через внешнюю таблицу
- Использует подмену партиций для минимизации блокировок

**Стек**: Greenplum, External Tables, PL/pgSQL, Partition Management

**Результат**:
- Загрузка 500К+ строк за ~90 секунд вместо полной перезагрузки
- Возможность параллельной загрузки нескольких партиций

### 3. `GP_udf_mart` — создание UDF для сборки и обновления витрины данных, уровень детализации магазин × день, обновление данных осуществляется по методу delta partition.

**Проблема**: Витрина `mart_store_day` (уровень магазин × день) собиралась вручную, что приводило к расхождениям и задержкам в отчётности.

**Решение**: Разработала функцию `f_upd_mart()`, которая:
- Автоматически агрегирует данные из факт-таблиц (bills_head, bills_item) и справочников
- Поддерживает инкрементальное обновление по дате (`calday`)
- Включает базовые проверки на полноту данных

**Стек**: Greenplum, SQL Aggregations, Window Functions, PL/pgSQL

**Результат**:
- Витрина обновляется автоматически в рамках Airflow-пайплайна
- Время сборки сократилось с 20 минут ручного SQL до 2 минут автоматического запуска
- Исключены человеческие ошибки при формировании отчётности

## Проекты в ClickHouse:

### 1. `CH_shards_replicas` — создание реплицированной, распределенной (шардированной) таблиц (масштабирование загружаемых данных).

**Проблема**: Рост объёма данных требовал горизонтального масштабирования без потери отказоустойчивости.

**Решение**: Спроектировала схему с:
- Репликацией на уровне таблиц (`ReplicatedMergeTree`)
- Шардированием по ключу (`Distributed`-таблица)
- Настройкой макросов для динамического определения шарда/реплики

**Стек**: ClickHouse, ReplicatedMergeTree, Distributed Engine, Macros

**Результат**:
- Возможность горизонтального масштабирования при росте данных
- Отказоустойчивость: при падении одного шарда данные доступны с реплики
- Готовый шаблон для создания новых шардированных таблиц

### 2. `CH_dictionaries` — создание словарей

**Проблема**: Частые JOIN'ы с небольшими справочниками (валюты, регионы) замедляли аналитические запросы.

**Решение**: Настроила словари (`Flat` / `Hashed`) для:
- Мгновенного подтягивания справочной информации
- Кэширования в памяти
- Автоматического обновления по расписанию

**Стек**: ClickHouse, Dictionaries (Flat/Hashed), XML Config

**Результат**:
- Ускорение запросов со справочниками на 30–50%
- Снижение нагрузки на CPU за счёт кэширования
- Готовый конфиг для добавления новых словарей

## Проекты в Oracle:

### 1. `ORA_proc_time` — функция по определению времени выполнения запроса (фрагмент).

**Проблема**: При оптимизации сложных запросов не было удобного способа замерить время выполнения отдельных участков кода.

**Решение**: Создала PL/SQL-блок с использованием `DBMS_UTILITY.GET_TIME`, который:
- Замеряет время выполнения запроса в сотых долях секунды
- Выводит результат через `DBMS_OUTPUT`
- Легко встраивается в любой анонимный блок для профилирования

**Стек**: Oracle PL/SQL, DBMS_UTILITY, Performance Profiling

**Результат**:
- Быстрая локализация "узких мест" в запросах
- Объективная метрика для сравнения версий запросов до/после оптимизации
- Используется при рефакторинге отчётных процедур
